Protégé MCP Server

An MCP (Model Context Protocol) server for Protégé Desktop, packaged as a single OSGi plugin. It exposes the live, active ontology of a running Protégé to MCP clients (Claude Code / Desktop, Codex CLI, VS Code MCP, ...) over a localhost HTTP endpoint. Reads and edits flow through Protégé's shared OWLModelManager, so they appear in the GUI immediately and join the undo stack - exactly like manual edits.

Requirements

Protégé must run on a Java 17+ JVM: the MCP Java SDK and the embedded Jetty 12 host are Java 17 bytecode, and the OSGi manifest declares Require-Capability: osgi.ee ... JavaSE 17. Select a 17+ JDK with the PROTEGE_JAVA_HOME environment variable before launching Protégé.

What you get

  • A MCP Server view (start / stop, the bound URL and bearer token) and a Settings > MCP tab.
  • 85 tools across explore / search, edit, curation, governance, documents, rules, history, reasoning, SPARQL, SHACL, project QC, impact analysis, release preparation, and audit export - every model edit goes through OWLModelManager.applyChanges, so it appears in the GUI and joins the shared undo stack (document open/save use Protégé's own load/save APIs).
  • One shared MCP endpoint for every Protégé window and instance - a small broker process owns the configured port and routes each MCP session to a live window.
  • Local HTTP (loopback by default; a bind-address preference can expose another interface) with OAuth 2.1 (dynamic registration + browser consent, persisted across restarts) and a static bearer token, supported in parallel.

v0.5.0

  • Shared MCP broker across Protégé windows and instances (default on; Settings > MCP toggle) — the configured port now belongs to a small broker process spawned on demand from the plugin jar, so one fixed URL (http://127.0.0.1:8123/mcp) works no matter how many windows or instances are open. Each MCP session stays pinned to one window; GET /instances lists the registered windows and /instances/{id}/mcp targets one explicitly. The broker exits by itself with the last instance, stays loopback-only by default (see the bind-address preference below), is kept a singleton by a cross-process file lock even when the configured port is ephemeral or held by a foreign app, and the plugin degrades to the previous standalone behavior automatically if the broker cannot be spawned or reached. The broker also runs from a staged copy of the plugin jar (~/.protege-mcp/jars/), so plugin updates never collide with a jar a live broker holds open.
  • New bind-address preference (Settings > MCP; default 127.0.0.1, presets ::1 and 0.0.0.0) for the client-facing server and broker; a red warning appears for non-loopback addresses (plain HTTP on your network), and OAuth authorization stays same-machine only whatever the bind — remote clients authenticate with the static bearer token.
  • The MCP Server view now surfaces a broker outage as Broker is down — press Start to relaunch it (Stop disables, Start relaunches the broker immediately); a stop made with the Stop button stays stopped — no auto-start overrides it — and reads as a plain stopped, never as an error.
  • A replacement broker no longer loses its takeover to the broker it just retired: a booting broker keeps retrying the singleton lock while the old one finishes dying, and a stopping broker releases the lock the moment it stops serving — previously the window could fall back to a standalone server on the configured port and split later instances onto an ephemeral broker.
  • The broker's idle linger is configurable (Settings > MCP, "Broker idle linger (seconds)"; default 15, range 0–3600) — how long the broker outlives the last Protégé instance so a quick restart reuses it and its port. A change reaches a running broker within a heartbeat while a window is attached; 0 makes the broker exit immediately with the last instance, so every quit-and-relaunch respawns it (MCP clients briefly see connection errors during that gap).
  • Dead MCP client registrations clean themselves up — no more revoking rows by hand after a client reconnects: a re-registered client's completed authorization drops the same-name registrations it replaced, abandoned registrations disappear after an hour, and clients silent for 60 days are removed with their tokens — in standalone mode and in the shared broker alike. Refreshing a token whose client record is gone now fails closed, so the client re-registers instead of looping on 401 errors.
  • The Ontology Assistant renders replies as Markdown (headings, lists, code, GFM tables; http(s) links open in the browser after a confirmation naming the real destination), and its "Show reasoning" toggle now actually streams the model's reasoning (the CLIs only send it when asked; each turn now asks).
  • Assistant replies can be copied as their original Markdown (styled rendering is lossy, so selecting and copying yields plain text): a copy button under each turn's closing reply puts the untouched markup on the clipboard, and a right-click menu offers "Copy" and "Copy message as Markdown" for any assistant message under the pointer.
  • Failed Ontology Assistant turns read cleaner: transcript error lines drop the [error] prefix (the red styling already marks them), and the generic claude exited with code 1 line no longer repeats a failure the stream already surfaced (e.g. a provider-side safeguard refusal) — it still appears when the CLI dies without emitting its own error, where it is the only diagnostic.
  • A second Protégé window or instance no longer loses the MCP server — and with it the Ontology Assistant — to Failed to bind: the server falls back to an ephemeral port when the configured port is busy, re-claims the configured port once it frees up, and concurrent live servers keep their OAuth / bearer-token state isolated (regenerating the token invalidates it on every live server immediately).
  • Guided prompts 6 → 11: new author_competency_question, author_swrl_rule, refactor_entity_safely, bootstrap_ontology, and release_readiness_check; the existing six are refreshed against the 0.4.x tool surface.
  • The competency-question annotation vocabulary (cq#competencyQuestion) is now published at its namespace IRI (vocabulary page + machine-readable cq.ttl).
  • Long help texts under Preferences > MCP and Preferences > Ontology Assistant now wrap instead of stretching the Preferences dialog into horizontal scrolling, and the Port / CLI-path labels sit next to their fields instead of floating at the dialog's right edge.
  • Tool descriptions are ontology-neutral: the MCP tool and parameter text illustrates its examples with neutral ex: / example.org placeholders instead of one ontology family's vocabulary, and deprecate_entity documents its default "term replaced by" property (IAO_0100001) as the de-facto OBO Foundry obsolescence convention rather than a standard.

v0.4.2

  • Reliability and authoring-ergonomics fixes from a full multi-module ontology reconstruction functional test, plus create_properties (batch property creation) — 64 → 65 tools.
  • run_reasoner now surfaces a failed classification (e.g. HermiT rejecting a SWRL built-in atom) as an error instead of a benign-looking Null-reasoner result; save_ontology save-as preserves the prefix map; side-effect entities get a Declaration (fixing spurious OWL 2 DL / undeclared-entity findings); run_qc_suite classifies for its reasoner stage; compound Manchester expressions accept CURIEs and IRI fragments; sub_property_chain_of accepts an inverse(P) link; and governance stops flagging standard/tool-internal vocabulary as owned.

v0.4.1

  • Modularization, batch intake, pagination, a SPARQL snapshot cache, and SHACL validation (61 → 64 tools).
  • extract_module: signature-based locality module extraction (the interactive analogue of robot extract) via the OWL API's SyntacticLocalityModuleExtractor — seed a signature and pick STAR (default), BOT (what the seeds use) or TOP (what uses the seeds) over the active ontology or its imports closure; load the module as a new workspace ontology or save it to a file. The extraction runs off the UI thread; no new dependency.
  • create_terms: batch term-request intake — the array form of create_term, applied as ONE undoable transaction with shared namespace/definition_property defaults. Atomic (a malformed or duplicate-IRI term aborts the whole batch) and strict-guarded.
  • Paginated reads: list_classes / search_entities / get_axioms_for_entity take an offset and return next_offset (totally-ordered sorts, so paging never drops or repeats an item) — enumerate a large signature past one page instead of a single truncated blob.
  • SPARQL snapshot cache: sparql_query reuses an edit-versioned snapshot instead of copying + serialising + re-parsing the imports closure on every call; any edit (or a set_prefix) transparently invalidates it, and each query still parses fresh immutable bytes so nothing mutable is shared across threads.

v0.4.0

  • Safe, testable LLM-assisted authoring (55 → 61 tools): close the propose → ground → verify → confirm loop and add a re-runnable requirements suite — all built by reusing shipping primitives (single-undo apply, the embedded reasoner, Jena ARQ, OWLEntityFinder, the catalog sidecar pattern).
  • apply_changes gains verify=report|rollback: classify the reasoner after applying and detect a regression caused by the batch (a newly unsatisfiable class, or a newly inconsistent ontology); rollback reverts the whole batch in one undo. Runs under a server-level write mutex; an intervening GUI edit degrades to report.
  • search_entities is now grounding-aware: each hit carries a score and match_kind (exact / prefix / substring / fuzzy), and the result adds best_match and would_mint so an assistant grounds a term before minting a new one. Results are now ranked (a behaviour change from the old alphabetical order).
  • A re-runnable competency-question suite (add_competency_question / list_competency_questions / remove_competency_question / run_competency_questions): pair a SPARQL query with an expected result and re-check them all against one shared snapshot, across three storage conventions (a ROBOT-style cqs/ *.rq folder, a full-fidelity JSON manifest, or ontology annotations).
  • verify_ontology: run project SPARQL invariants (forbidden patterns) with a fail_on severity gate. run_qc_suite: one umbrella gate composing reasoner + profile + structural + invariants + competency-question checks over one snapshot; absent backends are skipped with a reason, never an error.

v0.3.3

  • Ontology-development hardening (50 → 55 tools): project-governance validation, high-level curation macros, broader reasoner explanations, and a headless end-to-end smoke test.
  • New validate_governance tool: audit the active ontology against project policy — OWL 2 profile conformance (DL/EL/QL/RL, checked off the UI thread), an IRI policy (required namespaces / pattern), a required annotation suite (incl. label / definition), and module ownership / import layering (the module must not re-axiomatise imported terms).
  • New curation macros that each apply as one undoable step: create_term (class + label + definition + annotation suite + parents/restrictions + equivalents), create_property (property + domain/range/super-properties/characteristics/inverse), deprecate_entity (owl:deprecated + “term replaced by” pointer), and move_class (reparent a class, its subtree following).
  • get_explanations now handles any axiom type: for a kind it cannot minimally justify it falls back to confirming entailment and returning the asserted axioms that mention the same entities as structural context. validate_ontology gains a timeout_ms budget, and the preview_changes description now points at apply_changes.

v0.3.2

  • SPARQL support (47 → 50 tools): author, validate and run queries over the active ontology.
  • New sparql_query tool: run a SPARQL 1.1 query (SELECT / ASK / CONSTRUCT / DESCRIBE) over the active ontology and its imports closure, using an embedded Apache Jena ARQ engine. It is read-only — UPDATE and SERVICE are rejected, so a query never edits the model or reaches the network. Prefixes declared in the ontology (plus rdf/rdfs/owl/xsd) are auto-prepended, so queries need no PREFIX lines; limit caps the rows/triples returned. By default the query sees the asserted triples (like Protégé's SPARQL Query tab); set include_inferred=true to first materialise the active reasoner's inferences (run run_reasoner first).
  • New sparql_schema tool: the queryable vocabulary in one call — the prefix map (and a ready-to-paste PREFIX block), classes, object/data properties (with their domains and ranges), individuals and datatypes (each with a CURIE and full IRI), plus example queries built from the ontology's own terms. Use keyword to focus on a sub-topic.
  • New sparql_validate tool: check a draft query before running it (parse-only, or dry_run for a small sample). Reports whether it parses, the query form, the variables, whether sparql_query would accept it, and unknown_terms — IRIs used in the query that are not declared in the ontology (a typo / wrong-vocabulary detector).
  • New author_sparql_query MCP prompt that chains these: discover the vocabulary, draft, validate, run, iterate.

v0.3.1

  • The Ontology Assistant input now takes attachments. Long pasted text is compacted to a [Pasted content #N: ... chars] placeholder (the full body still reaches the assistant), and the Attach button, drag/drop, or clipboard paste add files and images as placeholders such as [Image #1] / [File #1: name].
  • Attachments are handled privately: each file/image is copied into its own temporary folder and only that single-file copy is exposed to the CLI — never the rest of its containing folder — and the copies are deleted when the turn finishes. The one-time egress banner now names attachments and pasted content (and is shown once more so the wider disclosure is acknowledged).
  • A placeholder you edit away before sending is reported and not sent, so nothing is silently dropped. Pressing Backspace next to a placeholder removes the whole [Image #N] token (and its attachment) at once.
  • Composer layout: + (bottom-left) attaches files/images; the send button sits at the bottom-right and is replaced in place by a stop button while a turn is streaming.

v0.3.0

  • In-Protégé chat assistant (Architecture Approach B). A new Ontology Assistant tab and view let you converse with an assistant that reads and edits the live ontology. Rather than calling a model API directly, it drives a locally-installed coding-agent CLI — Claude Code (claude) or OpenAI Codex (codex) — pointed back at this plugin’s own MCP server, so edits appear in the GUI and join the undo stack, and the read-only / confirm-write gates still apply.
  • Pick a provider with Use Claude / Use Codex (only installed CLIs are shown); the model picker is populated from the active provider. No API key is stored by Protégé — each CLI uses your existing login; a one-time banner discloses that prompts and ontology content are sent to your model provider via the CLI.

v0.2.2

  • Full-fidelity SWRL rule editing (41 → 47 tools): list_rules / add_rule / remove_rule read, add and remove SWRL rules (swrl:Imp) as structured body/head atoms, preserving named rule-variable IRIs (e.g. ex-var:process1) and rule-level annotations that a ?x text syntax cannot round-trip.
  • create_ontology mints a new empty module in the workspace and makes it the active edit target (pairs with set_ontology_id), so a multi-module ontology can be built from nothing.

  • diff_ontologies compares two ontologies (or the active ontology against a document) at the axiom level — a round-trip safety net that reports identical=true when a reconstruction is axiom-for-axiom faithful.

v0.2.1

  • apply_changes applies a previewed batch of axiom add/remove operations in one call and one undo entry, reporting per-operation results and any new entities introduced; strict=true refuses to mint a brand-new entity from an unrecognized IRI/name (a signal now surfaced by every write tool).
  • set_label upserts an rdfs:label; set_prefix registers a prefix; create_class / create_entity gain namespace / label / label_lang / no_label for term-namespace and language-tagged labels.

v0.2.0

  • Orientation & safety tools: get_ontology_context / get_entity_context summarise the ontology or a single term's neighbourhood; preview_changes shows an edit's diff (and any new entities) before it is applied; validate_ontology audits modelling quality (missing labels/definitions, duplicate labels, undeclared entities, properties with no domain/range, subclass cycles, …).

v0.1.2

The document is fetched off the UI thread (so a slow remote fetch never freezes Protégé) and wired in with Protégé's own copy-ontology / activate path — no modal load dialogs; not undoable.
  • Higher-level editing tools: rename_entity / delete_entity rewrite or remove an entity (and its referencing axioms) across the active ontology, undoable.

v0.1.1
